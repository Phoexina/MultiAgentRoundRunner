# MultiAgentRoundRunner

多 Agent 回合制循环执行框架，支持用户随时介入。将多个 AI Agent（Claude Code、Codex CLI、Gemini CLI）组织为可配置的 Pipeline，自动循环执行任务，并通过 Web 界面实时监控每个 Agent 的运行过程。

## 1. 系统概述

### 1.1 项目定位

MultiAgentRoundRunner 是一个**回合制多 Agent 自动化调度框架**，核心模式是：

- 将多个 AI Agent 配置为顺序执行的 **Pipeline**（如：Gemini 决策 → Codex 执行 → 用户确认）
- 以 **Round N-M** 为单位循环迭代，每轮可自动推进或等待用户确认
- 用户可**随时**向正在运行的 Agent 发送消息（用户指示），消息会在下一步骤的提示词中注入
- 提供 Web 界面实时监控每个 Agent 的流式输出、工具调用和执行状态

与传统"聊天机器人"的区别：用户不是主导对话的一方，而是**监督者/介入者**——系统自动运行，用户在需要时干预。

### 1.2 整体架构

```
┌─────────────────────────────────────────────────────┐
│                   浏览器 (Web UI)                     │
│  ConversationSidebar │ ChatUI │ PipelinePanel        │
└────────────────┬────────────────────────────────────┘
                 │ WebSocket (ws://host:port/ws)
┌────────────────▼────────────────────────────────────┐
│                  ChatServer (server.py)               │
│  HTTP 静态服务 + WebSocket 消息路由                   │
└────┬──────────────┬──────────────────────────────────┘
     │              │
     ▼              ▼
  Database      RoundController (round_controller.py)
  (db.py)         │
  SQLite          ├── _build_prompt()    提示词构建
                  ├── run_loop()         回合调度（批次并发）
                  └── ProcessManager (process_manager.py)
                        ├── AgentProcess: Claude CLI 子进程
                        ├── AgentProcess: Codex CLI 子进程
                        └── AgentProcess: Gemini CLI 子进程
                              │
                        stream_parser.py  实时流解析
                        session_reader.py JSONL 文件读取
```

### 1.3 技术栈

| 层 | 技术 |
|---|---|
| 后端语言 | Python 3.11+，全异步（asyncio） |
| WebSocket | websockets（legacy server API） |
| 数据库 | SQLite，aiosqlite 异步驱动 |
| 前端 | 纯 HTML / CSS / JavaScript，无构建工具 |
| Agent CLI | Claude Code（`claude`）、Codex CLI（`codex`）、Gemini CLI（`gemini`） |

## 2. 架构实现

### 2.1 进程模型

系统启动后是一个单 Python 进程，内部全异步驱动。Agent 以**子进程**方式启动：

```
Python 主进程 (asyncio event loop)
├── WebSocket Server Task
├── RoundController.run_loop Task
│     └── ProcessManager._active_processes (列表，支持并发)
│           ├── Claude CLI 子进程 (asyncio.subprocess)
│           │     stdout → stream_parser.parse_ndjson → 事件流
│           │     stderr → stream_parser.read_lines  → 错误流
│           ├── Codex CLI 子进程 (同上)
│           └── Gemini CLI 子进程
│                 stdout → stream_parser.parse_plaintext → 文本流
│                 JSONL 文件 → session_reader (轮询/事后读取)
└── _status_task (每 2s 广播进程状态)
```

`ProcessManager` 持有 `_active_processes` 列表，支持异步步骤并发运行多个 Agent。

**Windows 特殊处理**：使用 `asyncio.create_subprocess_shell`，终止时通过 `taskkill /F /T /PID` 递归杀死整个进程树。

### 2.2 核心模块职责

#### `main.py` — 启动入口

按依赖顺序初始化各组件：Database → ProcessManager → RoundController → ChatServer，将 `server.broadcast` 注入 `RoundController` 作为广播回调，启动 WebSocket 服务器。

#### `server.py` — WebSocket 服务器

同时承担两个职责：
- **HTTP**：通过 `process_request` hook 拦截 HTTP 请求，提供静态文件（`frontend/`）和 REST API（`/api/...`）
- **WebSocket**：管理 `clients` 集合，接收客户端消息路由到处理函数，通过 `broadcast()` 向所有客户端推送事件

#### `round_controller.py` — 流程调度核心

系统最核心的模块，负责：
- 维护 `RoundData`（当前 round/step 编号、pipeline 步骤、resume_id、user_inputs 等）
- `run_loop()` 按**批次**驱动 Pipeline 执行：同步步骤各自一批，连续异步步骤合并为一批并发执行
- `_build_prompt()` 拼装每步的提示词（注入历史、用户指示）
- 提供 `start` / `stop` / `stop_after_current` / `continue_next` / `start_new_round` / `resume_session` 控制接口
- `_single_step_only` 标志：`resume_session` 时设为 True，只跑该一步后停止，不写 DB 的 session_id 和 chat 字段

#### `process_manager.py` — Agent 子进程生命周期

`AgentProcess` 封装单个 Agent 子进程：
- `_build_claude_command()` / `_build_codex_command()` / `_build_gemini_command()`：构建 CLI 命令
- `start(delay_stdin=True)`：Gemini 专用，先启动进程再延迟写入 stdin，等 session_id 就绪后再发送提示词
- `stream_output()`：合并 stdout/stderr 为统一事件队列，逐条 yield `AgentOutput`
- `stop()`：触发进程树终止

`ProcessManager` 是对外接口，`start_agent()` 为异步生成器，在流结束后补发 `done` 事件。支持多进程并发（`_active_processes` 列表）。

#### `stream_parser.py` — 运行时流解析

- `parse_ndjson(stream)`：Claude/Codex 用，chunk 读取 stdout，逐行解析 JSON
- `parse_plaintext(stream)`：Gemini 用，逐行读取纯文本输出
- `read_lines(stream)`：逐行读取 stderr，返回原始字符串

#### `session_reader.py` — Session 文件读取

进程结束后从磁盘读取 Agent 写下的 JSONL 文件：
- **Claude**：`~/.claude/projects/**/{session_id}.jsonl`
- **Codex**：`~/.codex/sessions/**/*{thread_id}*.jsonl`
- **Gemini**：`~/.gemini/tmp/{session_id}/`（目录），内含对话 JSONL

Gemini 专用函数：
- `snapshot_gemini_sessions()`：启动前快照现有文件，用于区分新建 session
- `find_latest_gemini_session_id(pre_paths)`：轮询新出现的 session 目录，提取 session_id
- `find_gemini_session_by_time(created_at, working_dir)`：按时间戳匹配 Gemini session 文件（session_id 未记录时的兜底）

提供 `get_session_messages()`（结构化消息，供"查看运行日志"）和 `extract_session_summary()`（提取最后一条 assistant 消息，填充被中断 session 的摘要）。

#### `db.py` — 数据持久化

基于 aiosqlite 的异步 SQLite 封装，所有写操作通过 `_lock`（`asyncio.Lock`）串行化。

### 2.3 前端模块职责

前端为纯 JS，无框架，分为四个类：

#### `WSClient`

管理 WebSocket 连接与断线重连（指数退避，最大 10s）。重连后自动重新发送 `select_conversation` 恢复上下文。

#### `ConversationSidebar`

左侧边栏，管理：
- 会话列表（切换/新建/重命名/删除，通过弹窗 Modal）
- Session 列表（显示 Round N-M、Agent 类型徽章、运行状态徽章，分页，每页 20 条）
- 点击 Session → 触发 `ChatUI.openSessionFromSidebar()`
- 收到 `session_created` 时立即插入侧边栏（保留 pending 条目防止 DB 快照覆盖竞态）

#### `PipelinePanel`

右侧面板，可视化编辑 Pipeline：
- 每个步骤可选 Agent 类型（Codex / Claude / Gemini / 用户输入），可设异步并发
- Agent 步骤可自定义提示词模板；用户步骤无提示词（保存时强制置 null）
- 每个会话独立配置：working_dir、max_rounds（0=无限）、api_err_retries

#### `ChatUI`

中间主区域，双视图切换：
- **Chat 视图**：显示每轮摘要气泡（`user` / `user_input` / `codex` / `claude` / `gemini` / `system`），底部输入框发送消息，顶部控制栏
- **Session 视图**：实时显示单个 Session 的流式日志，标题栏有继续/刷新/删除按钮

Session 视图标题栏按钮：
- **继续**：弹窗输入提示词后发 `resume_session`（仅跑该步，不续后续）
- **刷新**：发 `refresh_session`，从 JSONL 文件重新提取摘要并更新 DB 和气泡

#### `App`

消息路由总线，将 WebSocket 消息分发给对应组件，维护组件间状态同步。

## 3. 数据存储

### 3.1 SQLite 数据库结构

数据库路径默认 `./data/sessions.db`。

#### `conversations` 表

每个"项目"对应一条记录：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER PK | 自增主键 |
| `name` | TEXT | 项目名称 |
| `created_at` | DATETIME | 创建时间 |
| `working_dir` | TEXT | Agent 工作目录（NULL 则用当前目录） |
| `max_rounds` | INTEGER | 最大轮次，0=无限 |
| `api_err_retries` | INTEGER | API 错误自动重试次数，默认 3 |

#### `pipeline_steps` 表

每个项目的 Pipeline 步骤配置：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER PK | 自增主键 |
| `conversation_id` | INTEGER | 所属项目 |
| `step_order` | INTEGER | 步骤顺序（0-based） |
| `agent_type` | TEXT | `'codex'` / `'claude'` / `'gemini'` / `'user'` |
| `prompt_template` | TEXT | 自定义提示词模板（NULL 时用全局默认；user 步骤始终为 NULL） |
| `is_async` | INTEGER | 1=与相邻异步步骤并行执行，0=顺序 |

#### `sessions` 表

系统唯一主数据表，同时存储 Agent 执行记录和用户消息：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER PK | 自增主键，也作前端 session_id |
| `conv_id` | INTEGER | 所属项目 |
| `round_index` | INTEGER | Round 编号（N，1-based） |
| `step_index` | INTEGER | Step 编号（M，1-based） |
| `agent_type` | TEXT | 见下表 |
| `session_id` | TEXT | CLI session UUID（用于 `--resume`，Agent 启动后异步写入） |
| `chat` | TEXT | 摘要文本；被中断时为 NULL |
| `created_at` | DATETIME | 创建时间 |

**`agent_type` 取值含义**：

| 值 | 含义 | 是否出现在侧边栏 |
|---|---|---|
| `claude` | Claude Code 执行记录 | ✓ |
| `codex` | Codex CLI 执行记录 | ✓ |
| `gemini` | Gemini CLI 执行记录 | ✓ |
| `user` | Pipeline 用户 step 的回应消息 | ✗ |
| `user_input` | 用户随时发送的指示消息 | ✗ |

侧边栏 `get_sessions` 查询过滤：`agent_type NOT IN ('user', 'user_input')`

### 3.2 Session JSONL 文件

**Claude** 路径：`~/.claude/projects/{encoded_dir}/{session_id}.jsonl`
每行事件：`system`（init）/ `user` / `assistant` / `result`（含 `is_error`）

**Codex** 路径：`~/.codex/sessions/YYYY/MM/DD/rollout-{datetime}-{thread_id}.jsonl`
每行事件：`response_item`，payload 类型：`message` / `function_call` / `function_call_output`

**Gemini** 路径：`~/.gemini/tmp/{session_id}/`（目录）
输出为 JSONL，每行含对话消息；stdout 仅为纯文本（不含结构化事件）

三种格式由 `session_reader.py` 分别解析，统一转换为：
```json
{ "role": "user|assistant|tool", "type": "text|tool_use|tool_result|thinking", "content": "...", "metadata": {} }
```

## 4. WebSocket 接口

连接地址：`ws://{host}:{port}/ws`

### 4.1 客户端 → 服务端

#### `user_message` — 发送消息

```json
{ "type": "user_message", "content": "消息内容", "conversation_id": 1 }
```

- 若当前 Pipeline 处于用户 step 等待状态（`is_waiting_for_input=True`）：存为 `agent_type='user'`，触发 Pipeline 继续
- 否则（随时发送的指示）：存为 `agent_type='user_input'`，下一步骤提示词的 `{user_input}` 注入该内容

#### `control` — 流程控制

```json
{ "type": "control", "action": "start|stop|stop_after_current|continue|start_round", "conversation_id": 1 }
```

| action | 说明 |
|---|---|
| `start` | 无历史时从 Round 1-1 开始；有历史时等同 continue |
| `stop` | 立即终止当前子进程，状态回 idle |
| `stop_after_current` | 当前步骤完成后停止 |
| `continue` | 从上次停止处推进到下一步骤 |
| `start_round` | 强制开始新的一轮（round_index +1，step_index 归 1） |


以指定 Session 的 CLI session_id 为起点用 `--resume` 继续运行。只跑该步骤本身，不续后续步骤，不更新 DB 的 session_id 和 chat 字段。

## 6. 核心功能实现

### 6.1 Round N-M 批次调度

**Round N-M** 表示第 N 轮第 M 步：
- N（round_index）：每当所有步骤跑完一遍，N 自动 +1
- M（step_index）：当前执行的是 Pipeline 中第几个步骤，1-based

**批次执行**：`_make_batches()` 将 Pipeline 步骤分组——连续的 `is_async=True` 步骤合并为一批并发执行，`is_async=False` 步骤各自单独一批。

```
Pipeline: [Gemini(sync), Codex(async), Claude(async), user(sync)]
Batches:  [[Gemini]]  [[Codex, Claude]]  [[user]]

Round 1: Gemini → (Codex ∥ Claude) → user → Gemini → ...
```

`run_loop()` 核心流程：

```
while True:
    for batch in batches[start_bi:]:
        results = await gather(*[_run_step(step) for step in batch])
        if api_error: 等待 60s → 重试失败步骤（最多 api_err_retries 次）
        if _single_step_only: break   # resume_session 专用，只跑一步
    if all done: round_index++, step_index=1, 继续下一轮
    if max_rounds reached: 停止
```

### 6.2 提示词构建

`_build_prompt()` 在每步执行前调用：

| 变量 | 内容 |
|---|---|
| `{round_number}` | 当前 round_index |
| `{chat_history}` | 最近若干轮窗口内的所有消息（含用户消息），oldest-first |
| `{user_input}` | 用户最新指示，无则为 `无` |
| `{last_summary}` / `{codex_summary}` | 历史中最后一条非 NULL 的 chat 摘要 |

模板支持 `{chat_history=N}` / `{user_input=N}` 语法指定窗口大小。

**Resume 时**：Agent 已有完整记忆，只注入 `user_input`（"继续"或用户输入的新提示词），不注入历史。

**用户指示优先级**：
1. `current_round.user_inputs`（本轮运行期间通过 WebSocket 发来，用完即清）
2. 若为空，查 DB `get_latest_user_message()` 取最近一条 `agent_type='user_input'` 记录

### 6.3 Agent 子进程启动命令

**Claude CLI**：
```
claude -p --output-format stream-json --include-partial-messages
       --verbose --permission-mode bypassPermissions [--resume {session_id}]
```
提示词通过 stdin 传入，输出为 NDJSON 流。

**Codex CLI**：
```
codex exec [resume {thread_id}] --skip-git-repo-check
      -s danger-full-access --json --config approval_policy="on-request" -- -
```
提示词通过 stdin 传入（`-` 从 stdin 读取）。

**Gemini CLI**：
```
gemini --yolo [--resume {session_id}]
```
提示词通过 stdin 传入。`--yolo` 自动批准所有工具调用（write_file、replace 等）。
stdout 为纯文本流（无结构化 JSON），完整对话写入 JSONL 文件。

启动时延迟写入 stdin（`delay_stdin=True`），先轮询等待 Gemini 新建 session 目录，获取 session_id 后再发送提示词。

### 6.4 Session 恢复机制

| 方式 | 场景 | DB 写入 |
|---|---|---|
| `resume_session(pk)` | Session 视图点击"继续" | 不更新 session_id / chat（`_single_step_only=True`） |
| `continue_next()` | 全局继续按钮 | 正常写入 |
| `start_new_round()` | 强制开新轮 | 正常写入 |
| API Error 自动重试 | 检测到 `result.is_error` | 正常写入 |

`resume_session` 专用逻辑：
- 设 `_single_step_only=True`，只运行被选中的那一步，结束后立即停止
- 不覆盖 DB 中原有的 `session_id` 和 `chat` 字段
- `session_pk` 复用原记录，不创建新 DB 行

### 6.5 摘要提取

**正常路径**：Agent 在输出末尾写入 `[SUMMARY]...[/SUMMARY]`，从事件流中提取。

**Gemini 专用**：Gemini stdout 为纯文本，不含摘要标记，始终从 JSONL 文件提取最后一条 assistant 消息作为摘要。

**兜底路径**：若正常摘要为空或为 fallback 字符串，`_run_step` 的 `finally` 块调用 `session_reader.extract_session_summary()` 从磁盘 JSONL 逆序查找最后一条 assistant 消息。

**手动刷新**：`refresh_session` 消息触发服务端重新从 JSONL 提取，更新 DB 并广播 `session_refreshed`，前端原地更新气泡内容。

### 6.6 Pipeline 用户 Step

Pipeline 中 `agent_type='user'` 的步骤不启动任何 Agent 子进程：

```
到达 user step
  → 检查 DB 是否已有该 (round, step) 的用户回应（服务器重启恢复用）
  → 若有：直接注入 user_inputs，继续执行
  → 若无：广播 system 气泡 "Round N-M · 用户"
         设 _waiting_for_input=True，等待用户发送消息
         用户发送后：存为 agent_type='user' 行，注入 user_inputs，继续执行
```

### 6.7 Session 日志视图

**实时模式**（Session 运行中）：监听 `session_log` WebSocket 消息逐条渲染。Claude text_delta 累积追加，最终 assistant 事件替换内容。Gemini 运行时开启文件轮询（每 2s 读 JSONL 文件刷新视图）。

**历史模式**（Session 已结束）：调用 `GET /api/sessions/{id}/messages`，从 JSONL 加载结构化消息，渲染完整对话（含工具调用、工具结果、思考过程）。

### 6.8 提示词模板变量速查

| 变量 | 说明 |
|---|---|
| `{round_number}` | 当前轮次编号（整数） |
| `{chat_history}` 或 `{chat_history=N}` | 最近 N 个正式步骤窗口内的全部消息（含用户消息），默认 3 |
| `{user_input}` 或 `{user_input=N}` | 最近 N 条用户指示，默认 1，无则为 `无` |
| `{last_summary}` / `{codex_summary}` | 历史中最后一条非空 chat 摘要（两者等价） |
